# Go CLI/TUI セキュリティ監査

Go製のCLIツールおよびTUIアプリに対するセキュリティ監査の段階的ワークフロー。自動ツールとAI支援を組み合わせて、機械的に検出可能な脆弱性から人間の判断を要する脆弱性まで、コストパフォーマンス順に進める。

## 全体の考え方

**自動ツールで取れるものを先に潰し、AIはその先の苦手領域に投入する。** このskillは以下の4フェーズで構成される:

1. **Phase 1: 自動静的解析** — `govulncheck` / `gosec` / `osv-scanner` / `go test -race` で既知CVEと典型的アンチパターンを機械的に潰す。AI不要、数分で完了。
2. **Phase 2: AI支援の脅威モデリング** — 外部入力の流入点→sinkまでのデータフローを網羅的に抽出し、attack surface を可視化する。
3. **Phase 3: ターゲット絞り込みコードレビュー** — TUI特有のエスケープ注入、`exec.Command` 安全性、ファイル操作、preview処理など、プロジェクト固有の危険箇所を攻撃者視点でレビュー。
4. **Phase 4: Fuzz強化** — パーサー等の堅牢性を `testing.F` で継続的に検証。

ユーザーの状況次第で特定フェーズだけ実施するのもOK。ただし Phase 1 は常に先に通すこと(ここを飛ばして高度な分析に行くのはROIが悪い)。

## ワークフロー

### 最初に確認すること

- プロジェクトのルートディレクトリのパス
- 監査対象の範囲(全体か、特定のパッケージ・機能か)
- TUI要素の有無(Bubbletea, tview, tcell等を使っているか)
- 外部プロセス呼び出しの有無(`os/exec`, `syscall` の使用状況)
- どのフェーズを実施するか(通常は1→2→3→4の順、部分実施もOK)

### Phase 1: 自動静的解析

`references/phase1-tools.md` を読み、記載されたツールを順番に実行する。結果を整理して、以下を報告する:

- 既知CVEヒット数とそれぞれの深刻度
- `gosec` 指摘のうち真の問題と誤検知の切り分け
- レース検出結果
- Phase 2 に持ち越すべき疑義(ツールが指摘しきれない・誤検知かもしれない箇所)

実行前に `go.mod` と `go.sum` の整合性確認、Go バージョン確認もしておくこと。

### Phase 2: 脅威モデリング

`prompts/threat-model.md` のプロンプトテンプレートに従い、プロジェクトを分析する。成果物は以下:

- **流入点の列挙**: argv, stdin, 環境変数, 設定ファイル, ネットワーク応答, 外部プロセス出力, ファイル内容など、信頼境界を越えて入ってくるデータの起点
- **sinkの列挙**: `exec.Command`, ファイル書き込み, ターミナル描画, ログ出力, HTTPリクエスト, デシリアライズなど、入力が危険な作用を起こしうる終点
- **データフロー**: 流入点とsinkの対応関係(どの入力がどのsinkに到達しうるか)
- **カバレッジ表**: Phase 1 の自動検出で既にカバー済みのもの / Phase 3 で手動レビューが必要なもの を分類

この段階では深追いせず、広く網羅することを優先する。深掘りは Phase 3 で行う。

### Phase 3: コードレビュー

Phase 2 で洗い出された危険箇所を、以下のreferenceと照合しながら詳細レビューする:

- **TUIを使っている場合**: 必ず `references/tui-checklist.md` を読む。ターミナルエスケープシーケンス注入は既製ツールが弱く、TUI fuzzy finder や preview機能を持つツールでは高リスク。
- **外部プロセスを呼ぶ場合**: `references/exec-safety.md` を読む。`exec.Command` の引数構築、`PATH` 解決、環境変数の扱い。
- **ファイル操作を多く行う場合**: パストラバーサル、symlink攻撃、TOCTOU、一時ファイル、パーミッション周り(このトピックは SKILL.md 内で扱うには広いので、必要に応じて別referenceを参照するか、個別に調査)。

プロジェクト固有の危険箇所があれば、`prompts/code-review.md` を使って攻撃者視点のレビューをClaude自身に依頼するプロンプトを組み立てる。この時、該当ファイルの内容を context に含めること。

### Phase 4: Fuzz強化

`references/fuzz-targets.md` を読み、fuzz対象として有望な関数を特定する。優先順位:

1. 外部入力を直接パースする関数(クエリパーサー、URL/パスパーサー、設定ファイルパーサー)
2. 外部プロセスの出力をデシリアライズする関数(`rg --json` の出力など)
3. ユーザー入力を加工してsinkに渡す関数

reference内のテンプレートに沿って fuzz target のスケルトンを生成し、初期corpusをseed する。

## 成果物の形式

各フェーズの完了時に、以下の形式でサマリーを出す:

```
## Phase <N> 結果

### 実施内容
- <実行したコマンド or 分析内容>

### 発見事項
| 優先度 | 項目 | 該当箇所 | 対策案 |
|--------|------|----------|--------|
| High   | ...  | file:line | ... |

### 次フェーズへの申し送り
- <次のフェーズで深掘りすべき疑義>
```

優先度は以下の基準で判定:
- **High**: 攻撃者が外部から悪用可能で、かつ秘密情報漏洩・RCE・権限昇格等に繋がるもの
- **Medium**: 悪用には特定条件(ローカルアクセス、特殊な入力等)が必要だが実害がある
- **Low**: ベストプラクティス違反、防御的には改善したいが単独では実害が薄いもの
- **Info**: 設計上の観察、現状問題ないが将来的に注意すべき点

## 重要な注意

このskillは**チェックリストとして機能する**が、**発見の網羅性は保証しない**。特にビジネスロジック起因の脆弱性や、そのプロジェクト特有のドメイン知識に依存する脆弱性は、汎用チェックリストでは拾えない。ユーザーにもこの制約を明示すること。

また、Phase 2-3 でAIが指摘した脆弱性候補は必ず**実際のコードを確認してから**報告する。Claudeが記憶ベースで「こういう脆弱性がありそう」と推測しただけの報告は、ユーザーの時間を浪費する。常に該当ファイルを `view` してから判断すること。
