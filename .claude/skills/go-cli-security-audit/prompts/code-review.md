# 攻撃者視点コードレビュー用プロンプトテンプレート

Phase 3 で、特定の危険箇所に絞って深掘りレビューをする際のプロンプト。Phase 2 で洗い出されたデータフローのうち「Needs manual review」判定のものに対して投げる。

## 使い方

1. レビュー対象のファイル/関数を特定する(Phase 2 の成果から)
2. 該当ファイルを context に含める
3. 以下のプロンプトテンプレートを状況に合わせて埋めて投げる

## プロンプト本体

```
以下のコードに対して、攻撃者視点でのセキュリティレビューを実施してください。

## レビュー対象

ファイル: <パス>
関数/メソッド: <関数名>
Phase 2 で特定された懸念: <データフローの概要>
想定される脆弱性カテゴリ: <例: ターミナルエスケープ注入、パストラバーサル>

## レビュー手順

### Step 1: 入力の理解

この関数に渡される `<引数名>` について、攻撃者がどこまで制御可能か分析してください:

- 呼び出し元を辿って、どの流入点から来るデータか特定
- 途中で施されるバリデーションや変換の有無
- 攻撃者が制御できる範囲(完全制御 / 部分制御 / 制御不可)

### Step 2: 処理の追跡

関数内部で、入力が最終的にどこ(どのsink、どの外部API呼び出し、どの描画経路)に到達するか、ステップバイステップで追跡してください。

### Step 3: 攻撃ペイロードの構築

想定脆弱性カテゴリに基づき、**具体的な攻撃ペイロード**を最低3つ考案してください:

- ペイロード内容(バイト列で)
- 期待される攻撃効果
- 現在のコードでそのペイロードが実際に通るかどうかの判定
- 通らない場合、どの防御が効いているか

### Step 4: PoC の書き方を提案

Step 3 のうち現在のコードで通るものがあれば、それを検証する単体テスト(PoC)のコードを提案してください。

### Step 5: 修正案

発見された脆弱性それぞれについて、以下を提示:

- 最小限の修正コード
- より根本的な対策(アーキテクチャ変更を伴うもの)
- 修正によって壊れる可能性のある既存の振る舞い

## 重要な注意

- 推測せず、実際にコードを読んだ結果として報告してください。
- 「安全です」と判定する場合は、なぜ安全なのか防御機構を明示してください。
- 実装が無い/不十分な場合も、誠実に報告してください(「フル実装されていない箇所があり、将来の実装次第でリスクになる」等)。
- コード外の要因(OS、ターミナルエミュレータの実装差、呼び出し側のコンテキスト)が関係する場合、それも明記してください。
```

## カテゴリ別の追加指示

レビュー対象の脆弱性カテゴリによって、Step 3 で試すべきペイロードが変わる。以下、カテゴリごとの追加指示例。

### ターミナルエスケープ注入の場合

```
Step 3 では以下のようなペイロードを試してください:

- `\x1b[2J\x1b[H` (画面クリア + カーソル原点移動)
- `\x1b]0;偽タイトル\x07` (ウィンドウタイトル書き換え)
- `\x1b]52;c;base64data\x07` (OSC 52 によるクリップボード書き換え)
- RTL override: `\u202E`
- ヌル文字: `\x00`
- 長大なエスケープシーケンス(DoS)
- UTF-8として不正なバイト列

これらが `<レビュー対象関数>` を通過して最終的にターミナルに到達するかどうか判定してください。
```

### パストラバーサルの場合

```
Step 3 では以下のようなペイロードを試してください:

- `../../../etc/passwd`
- `..\\..\\windows\\system32\\config\\sam` (Windows対応があれば)
- symlink指向: テスト中に `ln -s /etc/passwd victim` を作って入力として渡す想定
- 絶対パス: `/etc/passwd`
- NULバイト挿入: `foo.txt\x00.jpg` (古いCライブラリで問題になる、Go標準ではエラーになるはず)
- 長大なパス: `a/` を1万回繰り返す
- UTF-8 正規化: `\u2024` (ONE DOT LEADER, Unicode正規化で `.` と混同される可能性)

これらが `<レビュー対象関数>` を通過して、ベースディレクトリ外のファイルにアクセスできるか判定してください。
```

### コマンドインジェクションの場合

```
Step 3 では以下のようなペイロードを試してください:

- `;rm -rf ~;` (通常のシェルメタキャラ — exec.Command使用なら通らないはず)
- `--exec=payload` (オプションインジェクション — ツール側の `--` 区切り無しで通りうる)
- `-e` のようなフラグ混入(特に grep など、ツール依存)
- 長大な引数
- NUL文字を含む引数

特にユーザー入力が引数として渡る前の整形ロジック(文字列結合、エスケープ処理)に注目。
```

### ReDoS の場合

```
Step 3 では以下のような病的入力を試してください:

- 長大な繰り返し入力 (例: "a" * 10000)
- ネストされたパターン入力
- バックトラッキング誘発系 (Goの `regexp` は RE2 なので基本安全だが、`regexp/syntax` や独自マッチャーは別)
- `sahilm/fuzzy` のような独自マッチャーに対する病的入力

`time.After` でタイムアウト検知できる単体テストを PoC として提示してください。
```

## プロンプト例(実際の使用)

```
以下のコードに対して、攻撃者視点でのセキュリティレビューを実施してください。

## レビュー対象

ファイル: internal/tui/model.go
関数/メソッド: (m *Model) View()
Phase 2 で特定された懸念: rg --json 出力の `lines.text` フィールドが流入点から一切サニタイズされずに model.View() の描画に渡っている
想定される脆弱性カテゴリ: ターミナルエスケープ注入

[以下、共通の Step 1-5 + カテゴリ別追加指示]
```

こうして具体化したプロンプトを、対象ファイルをcontextに含めた状態で Claude に投げる。
